# うちレシピ 開発ルール（全モデル・全セッション共通）

自分専用レシピ帳PWA。設計書は `../docs/`（01企画〜08収益化ロードマップ）にあり、**着手前に必ず docs/06_開発バックログ.md を読むこと**。仕様判断に迷ったら 02_要件定義.md → 04_技術設計.md の順で確認。収益化・戦略に関わる判断は **docs/08_収益化ロードマップ.md** が一次参照文書（Fableの相談窓口がなくなった後の既定値集）。

## ユーザーについて
- プログラミング超初心者。報告は日本語で、専門用語は噛み砕いて説明する
- 大きな設計変更・お金が絡む判断は勝手にせず、ユーザーに確認する

## 相談先とエスカレーション（2026-07-07以降・Fable利用終了後の体制）

Fableは利用不可。収益化・戦略の既定値は **docs/08_収益化ロードマップ.md** にFableが策定済み。判断の優先順位:

1. ユーザーの指示（最優先。既定値と矛盾したらユーザーに従い、08を更新して履歴に残す）
2. docs/08 の既定値・決定表・権限区分（大半の「大きな判断」はここに答えがある。**勝手に再発明しない**）
3. それでも決まらないとき → 下記のOpus相談

**Opusサブエージェント相談（Agentツール, model: "opus"）を使う条件**（旧Fable条件を継承）:
1. ユーザーの感想・要望が曖昧で、複数の解釈がありうるとき
2. 要望が既存の設計・決定事項（docs/08の既定値を含む）と矛盾する、または変更を迫るとき
3. 同じバグ修正に2回失敗したとき
4. ユーザーが「Opusに聞いて」「相談して」と言ったとき（無条件で発動）
5. docs/08に既定値のない収益・戦略判断が発生したとき。**Opusには「選択肢2〜3個＋推奨1つ＋理由」を返させ、最終決定は必ずユーザー**（Fable時代の「Fableが決める」から「ユーザーが選ぶ」への転換）
6. docs/08でOpus推奨と明記されたバッチ（並行調理ナビ・栄養価概算・ストア化計画）の設計・実装

相談の作法（旧Fable運用を継承）: ユーザー発言は要約せず**原文のまま**渡す／試したことと結果（エラー原文）／関係ファイルパス／docsの該当箇所を添える。追加質問には新しいエージェントを立てず**SendMessageで同じエージェントに返信**して対話を続ける。決まったことは docs/README.md の決定事項ログか docs/08 に必ず記録する。

**ユーザー確認が必須（どのモデルも勝手に進めない）**:
- お金: 価格の決定・変更、販売の開始/停止、支出を伴う一切（ストア年会費・ドメイン等）、アフィリエイト導入
- 対外公開: note記事・X投稿の公開実行、LP/規約/プラポリ等の公開文書の新規公開や大幅変更、特商法関連
- 後戻りしにくいもの: 既存無料機能のPro化・削除、データを失いうる変更、50件制限のON
- レシピ内容の最終確定（管理栄養士ペルソナQAを通した上でユーザー承認、既存運用どおり）

**月次レビュー（Fableのフェーズ完了レビューの代替）**: 毎月初、計測数値・売上・ご意見箱をSonnetが表に整理し、docs/08のマイルストーンと照合して「順調/遅れ/要判断」を報告。要判断はOpus相談へ。手順詳細は docs/08 の「定常運用」参照。

## 解釈・実装時の運用ルール（2026-07-04 Fable事後レビューより）
音声入力フィードバックの解釈でミスが2件（曖昧な断片を誤って独立要望と解釈／位置指示より理由文の推論を優先）あったため、再発防止として以下を運用する。

- **A. 結合テスト**: 句点をまたいで結合すると自然な一文になる断片（例:「レシピ順」）は、単体で独立要望として立てない。分割/結合の両方の解釈を比較し、非可逆・データ設計に関わるものは確認、可逆で軽微なものは自然な方で実装した上で**完了報告の冒頭に解釈メモを明示**する
- **B. 指示は字義どおり**: 位置・文言・色など具体的指示と理由説明が併記されたら指示を字義どおり実装する。理由文から「より良い案」を推論して指示を上書きしない（理由は解釈が割れたときの判定材料としてのみ使う）
- **C. 2択提示は委任**: 「AするかBするかしてください」は確認不要でSonnetが選んでよい。選定基準は①可逆・非破壊な方②既存機能を妨げない方。選ばなかった案と理由をバックログに1行残す
- **D. 固定コンテンツ（スターターレシピ等）の文面編集**: ユーザーの具体的指示があればSonnet単独で編集可。ただし①変更後の文面を完了報告に原文のまま載せる②基本手順は変えずmemo追記を優先する③指示のない調理知識の新規書き下ろし、および加熱・衛生・保存に関わる変更はFable相談必須
- **D-④ 安全注記チェックリスト**（2026-07-04 Fable判断。新規レシピ・レシピセット制作時、該当したら必ず入れる。既存レシピへの遡及一括適用はしない＝QA・Fable相談で決まった分のみ反映する）:
  1. 肉・卵・魚を加熱するレシピ → 火通りの完了目安を該当stepのmemoに1文（色/肉汁/竹串。新規書き下ろし時は手順文に組み込んでも可）
  2. 半熟・とろとろ仕上げ → 「お子様・高齢者・妊娠中の方など」への完全加熱案内をレシピmemoに
  3. ルー物・煮物・汁物で4人前以上、または作り置き/翌日を勧める記述あり → 早めの冷蔵（大鍋は小分け）＋食べる前の沸騰再加熱をレシピmemoに
  4. 冷蔵保存前提の副菜（作り置きタグ） → 保存目安日数を明記
  5. お弁当タグ → 「完全に冷ましてから詰める」をmemoに（卵・肉が主材なら完全加熱も明記）
  6. 生の肉・魚をまな板/箸/素手で扱う → 交差汚染注意をmemoに
  - 文体: 簡潔な常体1〜2文。必須は「〜こと」、推奨は「〜と安心/〜がおすすめ」。菌名・恐怖表現・健康効能の誇大表現は書かない
  - **置き場所の原則**: doneness（その手順の完了判定）→該当stepのmemo（`s(text, minutes, memo)`の第3引数）／保存・再加熱・対象者・弁当・交差汚染→レシピ本体のmemo。**事後の追記でstepのtext本文を変更するのは不可**（唐揚げの温度目安はstep本文にあるが、これは新規書き下ろし時の設計であり事後追記の前例にはしない。事後追記は常にmemo）
- 上記いずれも「確認過多」に振れることを目的としない。バッチ運用（実装→検証→報告→ユーザーが違えば次バッチで訂正）は誤読を1往復で回収できる速さが利点であり、それを殺す過剰な事前確認はしない

## 環境
- Node.js は `~/.local/node` にある。**各Bashコマンドの先頭に必ず** `export PATH="$HOME/.local/node/bin:$PATH" && ` を付ける
- スタック: React + TypeScript + Vite + Tailwind v4 + Dexie(IndexedDB) + vite-plugin-pwa + lucide-react
- devサーバー: `npm run dev` → http://localhost:5173/ （ユーザーが開いていることがある。勝手に落とさない）

## コーディング規約
- UI文言はすべて `src/i18n/ja.ts` に集約（ハードコード禁止）
- 色は `src/index.css` のデザイントークン（CSS変数）のみ使用。直接色指定禁止。テーマは自動/ライト/ダーク/ブラウンの4種
- アイコンは lucide-react のみ
- スマホ縦画面基準・料理中に使う前提でボタン大きめ
- ルーティングは HashRouter。**basenameは付けない**（付けると白画面になる。過去に事故済み）
- 新フィールド追加時は任意項目にして既存データを壊さない（マイグレーション不要を保つ）

## 検証とコミット
- 完了条件: `npx tsc --noEmit` エラーゼロ ＋ `npm run build` 成功。ロジック追加時は簡易テストで動作確認
- コミットメッセージは日本語
- **push = 本番公開**（mainへのpushでGitHub Actionsが https://uchirecipe.github.io/ へ自動デプロイ）。動作検証が済んだ変更だけpushする

## 収益設計の前提（実装に影響する決定事項・2026-07-04 Fable最終改訂）
- 無料版=自分の登録50件まで（isStarter=true のスターター・配布セットは数えない）。制限はPro購入手段の公開と同一リリースでON（それまで FREE_LIMIT_ENABLED=false で寝かせる）。新規追加のみブロックし、既存データの閲覧・編集・削除とバックアップ復元は絶対に制限しない
- **バナー広告は導入しない**（技術的な審査可否ではなく、想定規模での収益がごく小さくUX毀損に見合わないため。独自ドメイン取得後も結論は同じ。詳細と再検討条件は docs/08 2-3-B）
- Pro=買い切りのみ（先行500円→大型更新時800円、既購入者据え置き）。解錠はオフライン検証のコード方式（docs/08 2-6。コード原本はリポジトリに絶対コミットしない）
- AI機能はサブスク禁止。AI回数券は無期限保留（自前サーバーを持つ判断をした場合のみ再検討）。写真取り込みはBYO-AI方式（ユーザー自身のAIでJSON化→貼り付け）で代替
- 個人データをサーバーに送らない（端末内保存）。計測は匿名アクセス統計（Cloudflareビーコン）のみで、これはプラポリに明記済みであること
- URLは https://uchirecipe.github.io/ （**2026-07にM1-0で独自ドメインへ一度だけ切替予定。切替完了後はこの行を新ドメインに書き換えること**）。**この一度きりの切替を最後にオリジン変更は禁止**（端末内データが見えなくなる）。詳細はdocs/08 2-3-A
- 決済・特商法表記・販売ページはアプリ外（BOOTH等）で完結させ、GitHub Pages側は紹介と外部リンクに留める（Pages規約対策、docs/08 2-7）
