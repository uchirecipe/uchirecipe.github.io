# うちレシピ 開発ルール（全モデル・全セッション共通）

自分専用レシピ帳PWA。設計書は `../docs/`（01企画書〜17、決定事項ログはREADME）にあり、**着手前に必ず docs/06_開発バックログ.md を読むこと**。**大きな判断・オーナー対応・レシピ制作の前には docs/17_Fable引き継ぎ書.md も読むこと**（オーナー解釈の判例集・予見分岐への既定値裁定・現在地。2026-07-09 Fable作成）。仕様判断に迷ったら 02_要件定義.md → 04_技術設計.md の順で確認。**アプリ開発（機能実装）のマイルストーン・進捗は docs/09_開発ロードマップ.md**、**収益化・価格・広告・決済・法務等の意思決定は docs/08_収益化ロードマップ.md** が一次参照文書。

## ユーザーについて
- プログラミング超初心者。報告は日本語で、専門用語は噛み砕いて説明する
- 大きな設計変更・お金が絡む判断は勝手にせず、ユーザーに確認する

## 相談先とエスカレーション

**重要判断の相談先は「そのとき使える最上位モデル」**（2026-07-08ユーザー決定）: 現在はAgentツールで**model: "fable"**。サブエージェント起動エラー等で使えなくなった場合のみ、下記の条件・作法はそのままに次点のモデル（model: "opus"）へ切り替える。収益化・戦略の既定値は **docs/08_収益化ロードマップ.md** にFableが策定済み。判断の優先順位:

1. ユーザーの指示（最優先。既定値と矛盾したらユーザーに従い、08を更新して履歴に残す）
2. docs/08 の既定値・決定表・権限区分（大半の「大きな判断」はここに答えがある。**勝手に再発明しない**）
3. それでも決まらないとき → 下記のFable相談

**Fableサブエージェント相談（Agentツール, model: "fable"）を使う条件**:
1. ユーザーの感想・要望が曖昧で、複数の解釈がありうるとき
2. 要望が既存の設計・決定事項（docs/08の既定値を含む）と矛盾する、または変更を迫るとき
3. 同じバグ修正に2回失敗したとき
4. ユーザーが「Fableに聞いて」「相談して」と言ったとき（無条件で発動）
5. docs/08に既定値のない収益・戦略判断が発生したとき。**Fableには「選択肢2〜3個＋推奨1つ＋理由」を返させ、最終決定は必ずユーザー**
6. docs/09_開発ロードマップ.mdでOpus/Fable推奨と明記されたバッチ（並行調理ナビ・栄養価概算・ストア化計画）の設計・実装

相談の作法: ユーザー発言は要約せず**原文のまま**渡す／試したことと結果（エラー原文）／関係ファイルパス／docsの該当箇所を添える。追加質問には新しいエージェントを立てず**SendMessageで同じエージェントに返信**して対話を続ける。決まったことは docs/README.md の決定事項ログか docs/08 に必ず記録する。**Fableの推奨内容は、ユーザーから聞かれていない部分も含めて、実装したか・後回しにしたか・却下したかを毎回もれなく報告する**。相談の回数・トークン量は気にせず、重要な判断は都度相談してよい。

**ユーザー確認が必須（どのモデルも勝手に進めない）**:
- お金: 価格の決定・変更、販売の開始/停止、支出を伴う一切（ストア年会費・ドメイン等）、アフィリエイト導入
- 対外公開: note記事・X投稿の公開実行、LP/規約/プラポリ等の公開文書の新規公開や大幅変更、特商法関連
- 後戻りしにくいもの: 既存無料機能のPro化・削除、データを失いうる変更、50件制限のON
- レシピ内容の最終確定（管理栄養士ペルソナQAを通した上でユーザー承認、既存運用どおり）

**月次レビュー（Fableのフェーズ完了レビューの代替）**: 毎月初、計測数値・売上・ご意見箱をSonnetが表に整理し、docs/09_開発ロードマップ.mdのマイルストーンと照合して「順調/遅れ/要判断」を報告。要判断はそのとき使える最上位モデル（現在はFable）への相談へ。手順詳細は docs/09 の「定常運用」参照。

## 解釈・実装時の運用ルール（2026-07-04 Fable事後レビューより）
音声入力フィードバックの解釈ミス（詳細はCLAUDE_archive_2026-07-08.md）の再発防止として以下を運用する。

- **A. 結合テスト**: 句点をまたいで結合すると自然な一文になる断片（例:「レシピ順」）は、単体で独立要望として立てない。分割/結合の両方の解釈を比較し、非可逆・データ設計に関わるものは確認、可逆で軽微なものは自然な方で実装した上で**完了報告の冒頭に解釈メモを明示**する
- **B. 指示は字義どおり**: 位置・文言・色など具体的指示と理由説明が併記されたら指示を字義どおり実装する。理由文から「より良い案」を推論して指示を上書きしない（理由は解釈が割れたときの判定材料としてのみ使う）
- **C. 2択提示は委任**: 「AするかBするかしてください」は確認不要でSonnetが選んでよい。選定基準は①可逆・非破壊な方②既存機能を妨げない方。選ばなかった案と理由をバックログに1行残す
- **D. 固定コンテンツ（スターターレシピ等）の文面編集**: ユーザーの具体的指示があればSonnet単独で編集可。ただし①変更後の文面を完了報告に原文のまま載せる②基本手順は変えずmemo追記を優先する③指示のない調理知識の新規書き下ろし、および加熱・衛生・保存に関わる変更はFable相談必須
- **D-④ 安全注記チェックリスト**（2026-07-04 Fable判断。新規レシピ・レシピセット制作時、該当したら必ず入れる。既存レシピへの遡及一括適用はしない＝QA・Fable相談で決まった分のみ反映する）:
  1. 肉・卵・魚を加熱するレシピ → 火通りの完了目安を該当stepのmemoに1文（色/肉汁/竹串。新規書き下ろし時は手順文に組み込んでも可）
     - **骨なしで厚みが不均一な部位（鶏むね肉の塊など）／骨付き肉（手羽先・手羽元等）は竹串+汁の色を使わない**（2026-07-06 Fable判断・バンバンジー/甘辛手羽先で確定）。竹串は深さが分からず、汁の色は温度の確実な指標でないため。代わりに「いちばん大きい・厚い1個を切り開いて（骨付きは骨のまわりを）目視し、赤み・生っぽさがなければ火が通った目安。赤ければ1分ずつ追加加熱」で統一する。厚み不均一なら加熱前に切り込みを入れて均一化する（骨付きは骨に届く深さの切り込みが均一化と確認窓を兼ねる）。温度計併記は初心者の所持前提が非現実的なため書かない。骨まわりの変色（骨髄由来）を「ピンクでもよい場合がある」と教えるのは危険側の誤読を生むため書かない（保守的に「赤ければ追加加熱」で統一）
  2. 半熟・とろとろ仕上げ → 「お子様・高齢者・妊娠中の方など」への完全加熱案内をレシピmemoに
  3. ルー物・煮物・汁物で4人前以上、または作り置き/翌日を勧める記述あり → 早めの冷蔵（大鍋は小分け）＋食べる前の沸騰再加熱をレシピmemoに
  4. 冷蔵保存前提の副菜（作り置きタグ） → 保存目安日数を明記
  5. お弁当タグ → 「完全に冷ましてから詰める」をmemoに（卵・肉が主材なら完全加熱も明記）
  6. 生の肉・魚をまな板/箸/素手で扱う → 交差汚染注意をmemoに
  7. はちみつを使う品 → 「はちみつを使っているため、1歳未満の乳児には与えないこと」をレシピmemoに(2026-07-11・第2弾フルーツヨーグルトバークで新設。加熱の有無を問わず必須)
  - 文体: 簡潔な常体1〜2文。必須は「〜こと」、推奨は「〜と安心/〜がおすすめ」。菌名・恐怖表現・健康効能の誇大表現は書かない
  - **置き場所の原則**: doneness（その手順の完了判定）→該当stepのmemo（`s(text, minutes, memo)`の第3引数）／保存・再加熱・対象者・弁当・交差汚染→レシピ本体のmemo。**事後の追記でstepのtext本文を変更するのは不可**（唐揚げの温度目安はstep本文にあるが、これは新規書き下ろし時の設計であり事後追記の前例にはしない。事後追記は常にmemo）
- 上記いずれも「確認過多」に振れることを目的としない。バッチ運用（実装→検証→報告→ユーザーが違えば次バッチで訂正）は誤読を1往復で回収できる速さが利点であり、それを殺す過剰な事前確認はしない
- **E. バッチ運用の追加ルール（2026-07-15 Fable事後レビュー。7/13〜7/15の摩擦点の再発防止）**:
  1. **前提を変える指示は本文まで掃引する**: 「全体的に◯◯前提で」のような枠組み変更の指示を受けたら、指示された箇所（memo等）だけでなく、**既存の手順本文がその前提と矛盾していないか全文を確認**する。複数品への一括適用前に**1品だけ直した見本を先に見せて方向確認**するとやり直しが1往復で済む（冷凍シリーズが3往復かかった反省）
  2. **見た目の感想を求める前に、オーナーの画面が最新か確定させる**: バッチ完了報告には「localhost:5173を再読み込みしてから確認してください」を必ず添え、テーマ・モックアップ等は**見る場所（URL/パス/設定手順）を明記**する。古い画面への感想で往復が発生した実例が2回ある（7/13夜・7/15朝）
  3. **デザイントークンを分割・新設したら旧トークンの全使用箇所を掃引**して「どちら側か」を分類する（--page導入時にsticky headerが白いまま残った反省）
  4. **外部の従量リソース（Codex等）は、消費モデル（無料枠・回復周期）を先に確定してから使う**。不明なら小さなpingで確かめてから本番の相談を投げる
  5. **サブエージェント（worktree便）のプロンプトには「ユーザーのdevサーバー(5173)を落とさない。プロセスをkillする場合は自分が起動したポート/PIDに限定する」を必ず含める**（pkill -f viteで巻き込み停止した実例）。また、オーナーの指摘リストを便に渡す前に**各項目が現状コードで既に解消済みでないか検証**し、済みの項目は便に含めない

## 環境
- Node.js は `~/.local/node` にある。**各Bashコマンドの先頭に必ず** `export PATH="$HOME/.local/node/bin:$PATH" && ` を付ける
- スタック: React + TypeScript + Vite + Tailwind v4 + Dexie(IndexedDB) + vite-plugin-pwa + lucide-react
- devサーバー: `npm run dev` → http://localhost:5173/ （ユーザーが開いていることがある。勝手に落とさない）

## コーディング規約
- UI文言はすべて `src/i18n/ja.ts` に集約（ハードコード禁止）
- 色は `src/index.css` のデザイントークン（CSS変数）のみ使用。直接色指定禁止。テーマは自動/ライト/ダーク/ブラウンの4種
- アイコンは lucide-react のみ
- スマホ縦画面基準・料理中に使う前提でボタン大きめ
- ルーティングは HashRouter。**basenameは付けない**（付けると白画面になる。過去に事故済み）
- 新フィールド追加時は任意項目にして既存データを壊さない（マイグレーション不要を保つ）

## ブランチ運用（2026-07-06導入）
- **`dev`が作業用の既定ブランチ**（GitHub上のdefault branchも`dev`）。日常の開発・コミット・pushはすべて`dev`で行う。`dev`へのpushは本番に一切影響しない（deploy.ymlは`main`へのpushのみに反応）ので、区切りのいいところで気兼ねなくコミット・pushしてよい
- **`main`は「本番にそのまま出す」意図のときだけ触る**。リリースするときは`dev`を`main`にマージ→`main`をpushする（このpushが本番デプロイのトリガー）
- 新しいセッションを開始したら、まず`git branch`で現在地が`dev`であることを確認する（`main`のまま作業を始めない）
- **現在、本番は準備中ページ表示中**（2026-07-08ユーザー指示。`main`のみ`MAINTENANCE_MODE=true`＝`src/logic/maintenance.ts`、`dev`はfalse）。**うかつにdev→mainへマージするとfalseが混入して意図せず再公開されるので、再公開はユーザーの明示指示があるまで行わない**

## 検証とコミット
- 完了条件: `npx tsc --noEmit` エラーゼロ ＋ `npm run build` 成功 ＋ **`npm test`**(ロジック単体・栄養回帰・レシピlintの一括実行、2026-07-08新設)。UI に触れた変更は **`npm run test:e2e`**(恒久Playwrightスモーク、devサーバーかpreviewを起動して実行)も通す
- **バグを直すときは、再発防止ケースを`scripts/test-logic.mjs`に足してから直す**(詳細はdocs/10 3章)
- コミットメッセージは日本語
- **`main`へのpush = 本番公開**（GitHub Actionsが https://uchirecipe.com/ へ自動デプロイ）。動作検証が済み、実際にリリースしたい変更だけを`main`にマージ・pushする

## 収益設計の前提（実装に影響する決定事項。最新・詳細はdocs/08が一次参照）
- **M4で販売するのは「無料版＋追加レシピパック（買い切り500円・2026-07-06正式確定）」の2商品のみ。Pro版はPvゲート（docs/09）発動まで発売しない**。実装済みPro機能（並行調理ナビ・月間献立・栄養概算）は`settings.proCode`ゲートで隠したまま削除も公開もしない。Pro価格はPro v2設計時に白紙で見直す（docs/08 2-8）
- 無料版=自分の登録50件まで（isStarter=true のスターター・配布セットは数えない）。**制限ONは追加レシピパック販売リンク公開と同一リリース**（それまで FREE_LIMIT_ENABLED=false で寝かせる）。新規追加のみブロックし、既存データの閲覧・編集・削除とバックアップ復元は絶対に制限しない
- **バナー広告は導入しない**（想定規模での収益がごく小さくUX毀損に見合わないため。詳細と再検討条件は docs/08 2-3-B）
- 解錠はオフライン検証のコード方式（Pro=`UR-`・パック=`UP-`。docs/08 2-6。**コード原本はリポジトリに絶対コミットしない**）
- AI機能はサブスク禁止。AI回数券は無期限保留。写真取り込みはBYO-AI方式（ユーザー自身のAIでJSON化→貼り付け）で代替
- 個人データをサーバーに送らない（端末内保存）。計測は匿名アクセス統計（Cloudflareビーコン）のみで、これはプラポリに明記済みであること
- **URLは https://uchirecipe.com/ 。これ以降オリジン変更は禁止**（端末内データが見えなくなる。詳細はdocs/08 2-3-A）
- 決済処理そのものはアプリ外（Stripe推奨・BOOTHも可）で完結させる。特商法表記は独自ドメイン`/about/`配下（Stripe選択時）またはBOOTH側（BOOTH選択時）のどちらでもよい（docs/08 2-5・2-7）
